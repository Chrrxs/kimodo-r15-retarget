# retarget — BVH (Kimodo/SOMA) → Roblox R15 animation retargeter

Converts BVH motion files generated by NVIDIA Kimodo on the SOMA `somaskel77`
skeleton (30 fps, centimeters, standard T-pose rest via `--bvh_standard_tpose`)
into Roblox R15 animations: a `KeyframeSequence` serialized as a `.rbxmx`
(Roblox XML model) file importable into Roblox Studio.

Pure Python 3 standard library — no dependencies. Tested with Python 3.12/3.14.

## Usage

```bash
python3 retarget/bvh_to_r15.py input.bvh --output out.rbxmx \
    [--rig r15]                          # target rig (only r15)
    [--scale-studs-per-meter 3.571]      # root-motion scale (default 3.571)
    [--fps N]                            # resample (default: keep source rate)
    [--keyframe-reduction-epsilon 0.5]   # degrees; 0 disables reduction
    [--loop]                             # KeyframeSequence.Loop = true
    [--priority Action]                  # Idle|Movement|Action|Action2..4|Core
    [--forward-axis z]                   # axis the BVH character faces: z|-z|x|-x
    [--name MyAnim]                      # default: input basename
```

Also writes `<out>.rbxmx.meta.json` with frame counts, duration, the joint
mapping used, and keyframe-reduction stats.

Run the tests (plain python, no pytest):

```bash
python3 retarget/tests/test_retarget.py
```

## The math

Conventions: quaternions are `(w, x, y, z)`; `a*b` applies `b` first
(`R(a*b) = R(a)·R(b)` on column vectors); BVH channels
`Zrotation Yrotation Xrotation` are intrinsic, `R = Rz·Ry·Rx` — identical to
what kimodo itself uses (`scipy Rotation.from_euler("ZYX", ...)`).

### Axis conversion

Both BVH/kimodo and Roblox are right-handed Y-up. The SOMA standard T-pose
faces **+Z** (eyes and toes at +Z; left arm along +X); Roblox characters face
**−Z**. The conversion `C` is a pure rotation about +Y taking the BVH forward
axis onto Roblox −Z (180° for the default `--forward-axis z`). It is applied
by conjugation to rotations (`C·q·C⁻¹`) and directly to translations. The
forward axis is configurable because it should be validated empirically in
Studio (see risks below).

### Rest-pose correction

Both rigs have "zero-rotation" rest poses (the standard-T-pose BVH stores
all-zero rotations; a Rig Builder R15 has all Motor6D `Transform`s at
identity), but the physical rest poses differ: SOMA holds a T-pose (arms
horizontal), R15 stands with arms hanging straight down.

For each mapped part we declare two unit bone-direction vectors in `r15.py`:
`src_dir` (SOMA rest, BVH axes — e.g. left arm `(1,0,0)`) and `tgt_dir` (R15
rest, Roblox axes — e.g. arm `(0,−1,0)`). The per-part correction is the
minimal rotation

```
D_part = quat_between(tgt_dir, C·src_dir)        # D·tgt_dir = C·src_dir
```

and each Motor6D transform is computed as

```
T_part(t) = D_parent⁻¹ · (C · L_src(t) · C⁻¹) · D_part
```

where `L_src(t)` is the composed local rotation of the collapsed SOMA chain
(e.g. `Spine1·Spine2·Chest` → UpperTorso) and `parent` is the parent part in
the R15 pose tree. Derivation: with per-joint world orientations
`W_tgt = W_src·D`, target bone directions track source bone directions
(`W_tgt·tgt_dir = W_src·C·src_dir`), and the Motor6D transform is the local
delta `W_tgt(parent)⁻¹·W_tgt(part)`.

Consequences (asserted by the test suite):

- Feeding the rest pose in yields `T = D_parent⁻¹·D_part`: **identity
  everywhere except the shoulders**, which get the expected ±90° roll about Z
  (`LeftUpperArm` −90°, `RightUpperArm` +90°) carrying the horizontal T-pose
  arms down into the R15 stance.
- Elbow/knee/wrist/ankle corrections cancel against their parent (same
  direction pair → same `D`), so they rest at identity.
- The minimal rotation leaves **twist about the bone axis unconstrained** — a
  canonical but not unique choice; arm twist may need per-joint tuning.

### Joint mapping (SOMA → R15 part / Motor6D)

| SOMA chain (rotations composed) | R15 part | Motor6D |
|---|---|---|
| `Root·Hips` | LowerTorso | Root |
| `Spine1·Spine2·Chest` | UpperTorso | Waist |
| `Neck1·Neck2·Head` | Head | Neck |
| `LeftShoulder·LeftArm` | LeftUpperArm | LeftShoulder |
| `LeftForeArm` / `LeftHand` | LeftLowerArm / LeftHand | LeftElbow / LeftWrist |
| `LeftLeg` / `LeftShin` / `LeftFoot` | LeftUpperLeg / LeftLowerLeg / LeftFoot | LeftHip / LeftKnee / LeftAnkle |
| (right side mirrors the left) | | |

Fingers, toes, eyes and jaw are ignored (R15 has no bones for them).

### Root motion

World hips position = `Root` position channels + `Root` rotation applied to
the `Hips` position channels (kimodo writes position channels as the full
local translation; in rest they equal the offsets). The delta from the
rest-pose hips position `(0, 100, 0)` cm is converted to Roblox axes and
scaled by `--scale-studs-per-meter / 100` (default 3.571 studs/m ≈ a 5-stud
R15 avatar vs a 1.7 m human), then written as the **LowerTorso** Pose
translation (the "Root" Motor6D transform) alongside the composed `Root·Hips`
rotation. The HumanoidRootPart Pose stays identity: Roblox's `Animator`
ignores translation on the HumanoidRootPart Pose, so the whole body moves
relative to the stationary HumanoidRootPart (verified in Studio 2026-07 with
a generated walk: ~5.5 studs/s of visible forward travel).

### Keyframe reduction

Greedy: a frame is dropped when slerp/lerp between the surrounding kept
keyframes reproduces every part's transform within
`--keyframe-reduction-epsilon` degrees (and 0.01 studs of translation for the
root). First and last frames are always kept.

warm-amber mannequin (capsules per bone) driven directly by the source BVH on
the left, and the classic blocky Roblox R15 avatar (real bundle meshes, see
`fetch_roblox_meshes.py`) driven by the retargeted world transforms

make fetch-meshes   # one-time: download the blocky R15 meshes from Roblox's CDN

Writes `<clipdir>/motion_compare_blender.mp4` (h264, 30 fps). Requires
Blender 4.2 LTS on PATH (or `BLENDER=/path/to/blender`) and imageio-ffmpeg
(auto-installed into `.venv` by the make target, or `pip install

`blender -b --factory-startup -noaudio --python ... --`; see its docstring).

works and is unchanged.

## Importing into Roblox Studio

1. In Studio, right-click in the Explorer (e.g. on `Workspace`) → **Insert
   From File…** → select the `.rbxmx`. A `KeyframeSequence` instance appears.
   (Dragging the file into the Studio 3D view also works.)
2. To turn it into a playable/publishable animation, the reliable route is
   the **Animation Editor**: open it on an R15 rig, and load the imported
   `KeyframeSequence` — the editor can read a `KeyframeSequence` placed under
   the rig's `AnimSaves` model (create a model named `AnimSaves` inside the
   rig and reparent the `KeyframeSequence` there, then it shows up in the
   editor's load list). From the editor you can inspect the motion and use
   **Publish to Roblox** to get an animation asset id.
3. Scripted alternatives exist for registering/previewing `KeyframeSequence`s
   at runtime, but their APIs have changed over time — verify against current
   Roblox documentation rather than trusting names here. The AnimSaves +
   Animation Editor route above is the safe, documented path.

## Known limitations / open questions for empirical validation

- **Forward-axis sign**: the T-pose file says SOMA faces +Z, and the default
  `--forward-axis z` maps that to Roblox −Z. If a test import walks backwards
  or is mirrored, try `--forward-axis -z` (and check left/right hand waves for
  mirroring — the conversion is a proper rotation so it should not mirror).
- ~~HumanoidRootPart vs LowerTorso root motion~~ **Resolved**: the Animator
  does ignore HumanoidRootPart Pose translation; root motion now rides on the
  LowerTorso Pose and plays back correctly (Studio-verified with a kimodo
  walk cycle: correct speed, direction, and foot alternation).
- **Forward-axis sign** is also confirmed: `--forward-axis z` (default)
  produced forward walking toward Roblox −Z with no mirroring.
- ~~Head tilted upward~~ **Resolved**: SOMA's rest neck (Neck1→Head) leans
  17.1° forward, so treating it as vertical pitched retargeted heads up ~20°.
  `_NECK_DIR` in `r15.py` now declares the true rest direction and the rest
  correction absorbs the lean (Studio-verified: head pitch ≈ −3° during the
  walk, i.e. level gaze). If another joint ever shows a constant angular
  bias, check its declared `src_dir` against the actual T-pose offsets first.
- **Bone twist**: rest corrections are minimal rotations; twist about the
  bone axis is a canonical choice. Arm roll may look off by a constant twist;
  fix by declaring per-joint twist offsets in `r15.py` if needed.
- **Proportions**: only rotations (plus root translation) are retargeted.
  R15 limb proportions differ from SOMA, so foot contacts can slide; no IK
  cleanup is performed.
- **Feet/legs**: SOMA's rest shin leans back ~5° and the ankle→toe vector
  dips ~21°; the declared bone directions treat both rests as "standing, feet
  flat, forward", so those small anatomical offsets pass through unchanged.
- **Easing**: all poses are written Linear/In; Roblox interpolates linearly
  between keyframes, matching the reduction error metric.
- **Priority tokens**: `Enum.AnimationPriority` values (Idle=0, Movement=1,
  Action=2, Action2=3, Action3=4, Action4=5, Core=1000) are hardcoded in
  `rbxmx.py`; verify Action2–4 if you use them.
